    Git常用命令

    阿星 2022/1/11

    # 新建本地分支

    • 新建一个和远程分支同名的本地分支

      git checkout --track origin/branch_name

    • 新建一个与远程分支不同名的本地分支

      git checkout -b new_branch_name origin/branch_name

    # 拉取代码

    • git pull

      等同于git fetch && git merge

    • git fetch

      只是拉取远程所有分支的更改，而不合并，合并推荐使用git rebase -i origin/xxxx

    # 分支相关

    • 新建本地分支，但不切换

      git branch xxx

    • 查看本地分支

      git branch

    • 查看远程分支

      git branch -r

    • 查看本地和远程分支

      git branch -a

    • 查看本地分支对应远程仓库的commit

      git branch -vv

    • 查看本地分支与远程分支的对应关系

      git remote show origin

    # 添加和提交暂存

    • 添加当前所有修改文件到暂存区

      git add

    • 提交暂存

      git commit -m 'xxx message'

    • 如果感觉提交的信息不好，可以修改

      git commit --amend

    # 暂存

    • git stash save '名称'（推荐）

      保存时给个标识，方便查找和恢复

    • git stash （不推荐）

      保存

    • git stash list

      查看所有保存的列表记录

    • git stash pop

      恢复最近保存的，不推荐使用

    • git stash pop stash@{num}

      恢复，num是可选项，通过git stash list查看具体数字，只能恢复一次

    • git stash apply stash@{num} （推荐）

      恢复，num是可选项，通过git stash list查看具体数字，可以恢复多次

    • git stash drop stash@{num}

      删除某个保存，num是可选项，通过git stash list查看具体数字

    • git stash clear

      删除所有保存

    结论：git stash的数据模型是基于栈的，并且存储栈是基于整个工作空间的，而不是某个分支的

    # git remote

    • git remote -v
      查看现有远端git地址

    • git remote set-url origin 远端地址

      将现有的远端地址替换为新的地址，如：git remote set-url origin https://xx.xx.xx.xxx/xxx/xx.git

    # 回滚git revert和git reset

    • git reset

      它的原理是通过改变HEAD的指向，改变到之前存在的某个版本，缺点：这个指向版本的后面版本都会被干掉

      1. 未使用git add .缓存的代码(工作区workspace)

        • git checkout --文件名

          如：git checkout --read.md

        • git checkout .

          放弃所有文件的修改

      2. 已经git add .缓存的代码(暂存区Index)

        • git reset 文件名

          如：git reset readme.md放弃指定文件的缓存

        • git reset .

          放弃本地所有缓存

        • git reset HEAD .

          放弃所有缓存

      3. 已经commit提交的代码(本地仓库)

        • git reset --hard HEAD^

          回到上一次commit的状态

        • git reset --hard 'commitId'

          回退到指定版本，commitId通过git log查看

    • git revert

      反做某一个版本，如commit了三个版本(版本1,2,3)，如果想撤销版本2的提交，又不影响版本3的提交，就可以反做版本2，生成新的版本4，版4里面包含了版3的内容，但是撤销了版2的东西

      场景：你的项目最近有2个版本要上线，这两个版本还伴随着之前遗留的 bug 的修复，一开始的时候，你将 bug 修复在了第一个版本的 release 分支上，突然在发版前一天，测试那边反馈，需要把第一个版本修复 bug 的内容改在第二个版本上，这个时候，第一个版本的集成分支提交应该包括了第一个版本的功能内容，遗留 bug 修复的提交和其他同事提交的内容，想要通过 reset 的方式粗暴摘除之前的关于 bug 修复的 commit 肯定是不行的，同时，这种做法比较危险，此时，我们既不想破坏之前的提交记录，又想撤回我们遗留 bug 的 commit 记录应该怎么做呢？git revert 就派上了用场。（所以针对每次提交的commit最好是解决一个问题或者一个功能需求，这样才能更好的回退）

      git revert <commit-id>     针对普通 commit
git revert <commit-id> -m  针对 merge 的 commit

    # 合并git merge/git rebase/git cherry-pick

    • git merge

      • 开发分支（dev）上的代码达到上线的标准后，要合并到 master 分支

        git checkout dev
git pull
git checkout master
git merge dev
git push -u origin master

      • 当master代码改动了，需要更新开发分支（dev）上的代码

        git checkout master 
git pull 
git checkout dev
git merge master 
git push -u origin dev

    • git rebase(合并提交记录和合并分支)

      特别注意，只能在自己使用的 feature 分支上进行 rebase 操作，不允许在集成分支上进行 rebase，因为这种操作会修改集成分支的历史记录。最好合并commit在push之前

      • 合并提交记录得到一个干净的提交历史（项目中用到了）

        1. 查看commit历史

          git log --oneline

        2. 查看是要合并最近提交的几条commit

          找到你要保留的那条 commit 的上一条 commit，复制这条 commit 的 id。记住一定要是目标提交的上一次提交记录，这样可以保证你要保留的那一条 commit 出现在 pick 清单的顶部，这非常重要。这里就需要根据你的实际情况来看了，3代表要合并最近的3条commit

          git rebase -i HEAD~3
# 或者
git rebase -i <base-commit>

          2.1 修改要保留的commit和要合并的commit

          会弹出一个vim的交互式界面，pick代表你要保留的那个commit，而squash代表要合并这些个commit，如果想合并多个，那么就把它们前面的pick都改为**s**

          注意：这里可能会出现一些合并有误等情况，需要根据实际情况看报错提示，一般会有下面这些：

          # 回到最初
git rebase --abort
# 编辑未完成的
git rebase --edit-todo
# 如果已经存在一个rebase-merge文件，可能需要删除
rm -fr ".git/rebase-merge"（双引号）

          2.2 此时会再次弹出一个弹窗，里面可以修改commit提交的文字描述，不需要的就可以注释掉，最后只留一个

          2.3 如果有冲突，解决了冲突，需要

          git add .
git rebase --continue

        3. 最后push到远程仓库（这里需要使用强制提交）

          git push origin xxx -f
# 或者
git push -f

      • 合并分支

        git rebase -i origin/xxxx

    • git cherry-pick

      当我们需要在本地合入其他分支的提交时，如果我们不想对整个分支进行合并，而是只想将某一次提交合入到本地当前分支上，那么就要使用 git cherry-pick 了

      git cherry-pick [commit-hash]
# 如果出现冲突，解决冲突后进行
git add 
git cherry-pick --continue

    #1、全局安装 nrm
npm install -g nrm

#2、增加源
nrm add bdr http://10.8.207.132:4873/

#3、切换源
nrm use bdr

#4、查看源
nrm ls

#5、删除源
nrm del <源名字>

#6、测试速度
nrm test npm

#7、可能需要先删除node_modules下的文件再进行
npm cache clean -f

# 安装不同版本的node
nvm install v14.xxx

nvm ls

nvm current

nvm uninstall <version>

    # package.json中依赖包前面的^是什么意思

    ^: 只会执行不更改最左边非零数字的更新。 如果写入的是 ^0.13.0，则当运行 npm update 时，可以更新到 0.13.1、0.13.2 等，但不能更新到 0.14.0 或更高版本。 如果写入的是 ^1.13.0，则当运行 npm update 时，可以更新到 1.13.1、1.14.0 等，但不能更新到 2.0.0 或更高版本。

~: 如果写入的是 〜0.13.0，则当运行 npm update 时，会更新到补丁版本：即 0.13.1 可以，但 0.14.0 不可以。

>: 接受高于指定版本的任何版本。

>=: 接受等于或高于指定版本的任何版本。

<=: 接受等于或低于指定版本的任何版本。

<: 接受低于指定版本的任何版本。

=: 接受确切的版本。

-: 接受一定范围的版本。例如：2.1.0 - 2.6.2。

||: 组合集合。例如 < 2.1 || > 2.6。
